# Theme-tagging cheat-sheet (Vibe Coding Governance SLR)

Tag on the **document contents**. **Multi-tag freely** вЂ” a paper can carry several themes.
Tag on **membership** (the paper *contributes to* that theme's argument), **not mention**.
Pick **one PRIMARY theme** (the paper's home / where it'd be written up in depth; tie-breaker and lit-review override below).
**Steering exclusion:** shaping what gets *generated* вЂ” prompts, specs, fine-tuning, **controlling the AI's inputs/context** вЂ” is steering, NOT oversight; tag only the paper's inspection/comprehension/gating contribution.
**Plumbing в‰  membership** (systems papers): tag only mechanisms the paper *argues about*, not everything its system happens to use (agents running test suites в‰  `rules-based-checks`).
**Documented practice counts:** empirical evidence of practitioners *exercising* a stage's mechanism earns membership in that stage's theme. **Vision-paper floor:** a stage mentioned in one scenario sentence = mention, not membership.

## THEME TAGS (`theme:<slug>`)

**Problem вЂ” quantify**
- `oversight-scaling-inversion` вЂ” AI code is riskier yet *less* inspected; PRs auto-merged unreviewed; review is the bottleneck; burden piles on maintainers
- `ai-code-insecurity` вЂ” empirical **security**-vulnerability evidence in AI code (CWEs, insecure-but-confident), **incl. hallucinated/poisoned dependencies in the output** (D4 ruling). Incident analyses & original vulnerability taxonomies count; secondhand rate citations alone don't
- `quality-debt` вЂ” **non-security** quality decay: tech debt, complexity, smells, maintainability, breaking changes

**Limits of current oversight**
- `automation-bias` вЂ” the *human* fails: over-trust, complacency, skill erosion, misses flaws even when warned. **Requires a CAPABLE human failing** (attention/trust); a human who *can't evaluate at all* (non-developer settings; failure survives priming+incentives) = competence gap, NOT bias (22JBEZNK). Competence-gap papers **stay core-eligible**: tag the remainder (explanation/mode/population facets) вЂ” the missing theme is a staged candidate, not a demote signal
- `oversight-theater` вЂ” review exists on paper but lacks authority/time/info to change the outcome (rubber-stamp, token HITL)

**Solution вЂ” the Detect в†’ Triage в†’ Fix в†’ Escalate pipeline** *(gray-zone detectors вЂ” rubric-grounded LLM-judge, "LLM writes tests then runs them" вЂ” expect dual-tagging of both Detect themes)*
- `ai-review` вЂ” [DETECT] AI *judges* the artifact (probabilistic, can hallucinate); incl. multi-agent / cross-model review, **its reliability limits**, and empirical evidence of practitioners *using* agents to validate agents' output
- `rules-based-checks` вЂ” [DETECT] deterministic **grounded** checks: tests, static analysis/lint, symbolic exec, sandbox, classical formal-methods engines
- `formal-methods` вЂ” [technique, COMPOSABLE] theorem proving / model checking / symbolic exec / autoformalization. Pair with the performer: AI does it в†’ `ai-review`+`formal-methods`; classical engine в†’ `rules-based-checks`+`formal-methods`; no performer (pure advocacy/position) в†’ `formal-methods` + the `intro-framing` *facet*
- `risk-routing` вЂ” [TRIAGE] the **allocation decision** вЂ” *which/whether/when* AI items reach a human & at what tier; the **smarts of surfacing** (signal + selection/tiering logic). NOT the human's control mechanism. **(= WHAT gets surfaced)** Error-condition handback ("agent stuck в†’ human") is NOT routing вЂ” no triage decision в†’ `hitl-workflow` / `remediation-gating`. Signal must be **computed & producer-independent** вЂ” model self-confidence is disqualified. **Defining a risk metric/score without the allocation decision is NOT routing** в†’ `metrics` (+ `routing-signal` if framed for review attention)
- `remediation-gating` вЂ” [FIX] governance of **autonomous** fixing (Jidoka: the system fixes *without per-fix human involvement*, kept safe by machinery) вЂ” **content** gates (filter fix candidates) and **process** gates (bounded retries / budget-decay / convergence / stop-progression, fail-closed); not the repair technique itself. **REQUIRES an autonomous fix being overseen** вЂ” a detect/publish gate with **no auto-fix** (VibeGuard) is NOT this (detector's enforcement side). Deciding when a **human** must engage on the fix path (risky fixes в†’ human) = `risk-routing` layered on top
  - *Anti-pattern note:* a human approving **every** fix is NOT remediation-gating вЂ” review-everything is the unscalable anti-pattern (a paper *advocating* it в†’ `counterpoint` candidate). Such a design *introduces* automation-bias risk вЂ” that's a critique to note, NOT `automation-bias` membership (membership requires the paper to *study* the human failure)
- `hitl-workflow` вЂ” [ESCALATE] the human's **control surface** вЂ” *how the human acts* once engaged: checkpoints, action guards, approval gates, bounded delegation. NOT what to surface. **Levers, not lenses** (comprehension/visibility tools в†’ `oversight-explanation`; a lever over the AI's *inputs* is steering). **Plan-gate rule:** a human gate over an AI-produced plan IS hitl-workflow when it's a **designed checkpoint in a lifecycle** (defined gate, authority to block); the same approval inside a conversational guide-then-"go do it" flow is steering. **(= HOW + WHEN the human acts)**
- `oversight-explanation` вЂ” [ESCALATEВ·info side] **help the human understand what the AI is doing** вЂ” any system-provided or human-invoked support for understanding/judging AI output. Modes (illustrative, NOT exhaustive): *push* = decision-ready escalation handoff (background + options + recommendation + risks); *pull* = human-invoked visibility into what the AI is doing/using; *standing* = explanations attached to AI output that support the human's verdict. Lens, not lever (lever в†’ `hitl-workflow`). **Explainability routes by object:** support for judging the AI/its output в†’ here; **comprehensibility of the code itself** (unclear logic, undocumented) в†’ `quality-debt` territory / `risk-quality`; model-XAI в†’ `general-ai`
- `agent-scope-drift` вЂ” [cross-cutting] agents wander off-mandate / make unreviewed decisions; mechanisms that **detect/bound departure from intent**. Tag by the mechanism's **object**, not the actor's motivation: a panel checking code scope vs spec scope в†’ here + `ai-review`; retained control merely *motivated* by drift-worry в†’ `hitl-workflow` only

**Worked decomposition (multi-stage systems вЂ” assign each mechanism by its object):** producer's own pre-submit checking/iterating = self-conformance, NO theme В· independent evaluation of the submitted artifact = detector (`rules-based-checks`/`ai-review` by epistemics; the *additional, producer-independent* checks are the oversight value; blocking on failure = the detector's enforcement edge) В· deciding which findings matter, e.g. by severity + cross-model agreement (a computed, producer-independent signal) в†’ `risk-routing` В· governing the ensuing **autonomous fix path** (filters, bounds, convergence, fail-closed stop вЂ” no per-fix human) в†’ `remediation-gating` В· deciding when a fix must engage a **human** (risk tiers on the fix path) в†’ `risk-routing`

**Governance & policy**
- `org-governance` вЂ” the *org-level governance apparatus* applied broadly: policy, **audit logging**, accountability, roles, maturity models, policy-as-code (internal; в‰  a single pipeline framework)
- `regulatory-compliance` вЂ” *external* law/standards: EU AI Act Art.14, NIST AI RMF, ISO 42001, liability, audit evidence

**Supply chain & provenance**
- `tooling-supply-chain` вЂ” provenance/vetting of the AI **tools** (skills/MCP/agents); **+ attacks on the reviewer**. **Dependency risk *in generated code*** (hallucinated/poisoned packages the AI writes in) в†’ `ai-code-insecurity`, NOT here (D4 ruling). Excl. keyword false-positives (hardware trojans, classic supply-chain incidents)
- `provenance-auditability` вЂ” traceability of AI **changes** so a human *can* review; auditable record; IP/licensing. Requires a **persistent record serving HUMAN reviewability/audit** вЂ” a point-in-time "what's in use now" view в†’ `oversight-explanation`; persistence serving only agent coordination = plumbing

## FACET TAGS (functional role; orthogonal вЂ” optional, apply if they fit)
- `problem-statement-anchor` вЂ” a single committee-sit-up empirical stat **anchoring the OVERALL problem statement** (the scaling inversion / two-part frame) вЂ” NOT a sub-argument's or population-specific headline number, however vivid. Never on `lit-review` (secondhand вЂ” anchor the primaries)
- `survey-input` вЂ” adoption / preference / RAI-priority finding useful for the org survey design; requires **substantive** findings вЂ” one incidental adoption stat в‰  membership
- `intro-framing` вЂ” position / agenda / definitional paper that *names the gap* but **doesn't operationalize a mechanism**
- `lit-review` вЂ” secondary literature (survey / review / meta-analysis) вЂ” **systematic OR narrative**; test = evidence *synthesized from other papers, not produced*. **A framing related-work section в‰  this facet** вЂ” the facet is the PAPER's role; a paper producing its own results with a review as framing is primary literature. **If `lit-review` applies, the primary MUST be the biggest-tent theme** = the theme under which the largest share of the synthesized evidence would be cited; ties в†’ the review's own framing question (never the most vivid/quantified section)
- `counterpoint` вЂ” [role facet] the paper **argues against a prevailing position** (automation-maximalism, HITL value, oversight scaling вЂ” any direction); note *what it opposes* in the rationale. Often pairs with `intro-framing`

**Artifact / evidence cluster** (composable; form в†’ maturity):
- `framework` вЂ” a **technical framework / reference architecture integratable into a build pipeline** (whether or not built). в‰  `theme:org-governance` (the org apparatus). A bare taxonomy/decision-model gets neither; **not a one-off point tool / bare point result** вЂ” a focused single-concern architecture qualifies (VibeGuard, Hedwig). Test: *would someone adopt it as a reusable pipeline design?* Composes: `framework`+`built-system`+`adopted`.
- `design-only` вЂ” the paper **specifies a mechanism in buildable detail** (components, formulas, thresholds, architecture) **but never credibly runs it**: no working implementation, no evaluation on the intended object. **A mock demo в‰  built** вЂ” a demo that fabricates the mechanism's core outputs (random/synthetic stand-ins for what it's supposed to compute) is still design-only (R4WJZBSF). Only for papers that *propose* something вЂ” an empirical study with no proposed mechanism gets nothing. Mutually exclusive with `built-system`/`adopted`; composes with `framework`. в‰  `intro-framing` (which specifies NO mechanism вЂ” discriminator = buildable detail; **a stated architectural-design contribution qualifies even when realization is deferred to a research agenda** вЂ” F9JM9CI6 ruling). Core bar: **what+how carve-out** вЂ” names the elements to track AND defines operationalizable metrics в†’ **clears the bar as measurement even unevaluated** (R4WJZBSF); what-only enumerations / borrowed-metrics / unevaluated frameworks *without* defined metrics в†’ context candidates. **Carve-out exclusion:** metrics auditing a measurement tool's/judge's OWN reliability = tool validation в†’ context (BAWCBT9R), not the carve-out
- `expert-validated` вЂ” the contribution was **evaluated by documented expert judgment** вЂ” panel / Delphi / structured practitioner review, with the process described as method (who, how many, what protocol). **The panel judges the *contribution*, not produces the *data***: experts shaping requirements or filtering lit-review inputs (input-side) don't count; experts as study subjects = `method-self-report`, not this; в‰  peer review; в‰  undocumented collegial feedback. Evidence rung: unvalidated design < expert-validated < built prototype < adopted; composes with `design-only` (the common case) or `built-system`. Can co-hold with `method-self-report` (panel provides data AND validates)
- `built-system` вЂ” the authors *implemented* it as a working system/tool/prototype ("and they built it")
- `adopted` вЂ” used **outside research** (commercial / production / real org use), beyond a lab prototype/benchmark. Scarce, high-signal; absence = prototype/proposal. **Pilot rule:** "outside the research *context*" в‰  "outside the research *org*" вЂ” a pilot/case study run *as the paper's field evaluation* (org = study site) evidences `built-system`, NOT adoption; `adopted` = use for the org's **own operational purposes**, continuing beyond the study (integrated/org-initiated/ongoing)

**Generation-mode scope pair** (which *setting* the paper studies; cut on **who initiates + the reviewable unit**, NOT tool location вЂ” agents live in IDEs too):
- `assistive` вЂ” **human-initiated, suggestion-granularity** generation (inline completion, **snippet-level** chat-paste): the human authors in the flow, accepts piece-by-piece вЂ” a chat task returning a *complete artifact* is the tie-rule case below. Oversight surface = the *acceptance moment*.
- `agentic` вЂ” **AI-initiated / AI-planned multi-step work** delivered at **artifact/PR granularity** for review. Oversight surface = the *gate*.
- Apply either or **both** (paper compares/spans modes); **neither** = mode unspecified / irrelevant to the paper's claim. **The pair describes the *generation* studied:** "uses agents" в‰  `agentic`, "AI assists the human" в‰  `assistive` вЂ” AI only on the review/oversight side в†’ **neither** + consider `general-code`. **Tie-rule:** initiator vs reviewable unit disagree (human-prompted chat task в†’ complete artifact) в†’ **the reviewable unit dominates** (wholesale artifact = the gate = `agentic`).

**Scope / contribution / population / risk-type flags:**
- `general-ai` вЂ” [scope flag] governance/oversight is **general AI/LLM, not coding-specific** (model robustness, general RAI frameworks). Default (untagged) = coding-specific. Flags candidates for context.
- `steering` вЂ” [contribution flag] the paper's proposed solution **or documented practice** operates on **generation** (prompts, specs, fine-tuning, shaping model inputs) rather than inspecting/gating the artifact, as a **substantive part of the contribution** вЂ” NOT any incidental prompt-shaping component every AI system has. **Contribution, not topic.** Steering-**only** solution в†’ demote candidate
- `metrics` вЂ” [contribution flag] the paper **defines metrics/scores/indices** (risk, quality, oversight, compliance) **as a deliverable** вЂ” not merely *uses* metrics to evaluate something (every empirical paper has an evaluation apparatus; that gets nothing). **Contribution, not apparatus.** The metrics' *object* comes from co-tagged themes (+`regulatory-compliance` = compliance-risk metrics; + an oversight theme = oversight metrics). Composes with `design-only` (defined, never run on real data вЂ” R4WJZBSF) or with problem themes (defined AND applied в†’ the findings earn the theme; the facet marks the reusable instrument). **Rationale MUST name the measurand** using the fixed vocabulary: security В· quality В· comprehensibility В· over-reliance В· ip-plagiarism В· bias В· compliance В· privacy. **Defining a metric for X does NOT by itself earn X's theme** (define-only в†’ `metrics` + the matching `risk-*` flag; the theme requires the lift вЂ” first catch: R4WJZBSF)
- `routing-signal` вЂ” [contribution flag] the paper contributes a **computed, producer-independent signal framed for review-attention allocation** (which artifacts deserve human scrutiny) **without operationalizing the selection/gating logic** (no threshold / tier / engagement decision). The audit trail for the signal-without-allocation boundary + the Triage supply-chain roster (signal suppliers vs operationalized routers). Usually composes with `metrics`. **Mutually exclusive with `theme:risk-routing`** (operationalized selection в†’ the theme, not this flag). Signals framed for other purposes get `metrics` only (R4WJZBSF's org-risk score) вЂ” the framing test keeps generic defect predictors out
- `non-developer` вЂ” [scope flag] the generating/overseeing human is **not a professional developer** (end-user / business user / citizen developer). Default (untagged) = professional devs
- `general-code` вЂ” [scope flag] the oversight/review mechanism targets **code generally, not AI-generated code** (AI may sit on the *review* side only); transfers to our setting but wasn't developed/evaluated there. Default (untagged) = the overseen object is AI code
- `risk-security` / `risk-quality` / `risk-overreliance` / `risk-ip` / `risk-bias` вЂ” [risk-type flags, uniform bar] **substantive treatment of the harm**: defines a metric for it В· contributes/evaluates a mitigation for it В· reports an empirical result about it В· devotes focal analysis. **An intro risk-list sentence = mention, NO tag** вЂ” rationale cites which clause fired. Types: security В· quality (incl. **code comprehensibility** вЂ” "explainability of the code") В· over-reliance В· IP/plagiarism/licensing В· bias in generated code. **Source-agnostic (CodeAgent ruling):** the flags track the HARM regardless of the overseen code's source вЂ” a `general-code` mechanism mitigating security/quality earns the flags + `general-code` (the matrix segments transferable mechanisms via that flag); object = the *model* (`general-ai`) still gets none. **Flag в‰  theme:** the flag marks substantive engagement (define-level suffices); the matching theme (`ai-code-insecurity` / `quality-debt` / `automation-bias`) still requires the paper to *do the lift* вЂ” heavy-lift papers carry **both** (expected co-occurrence). Matrix: identified/measured axis = these flags, one query, one bar. **Lit-reviews CAN earn these flags via focal secondhand synthesis** (2CKL96B8: security+quality) вЂ” the matrix segments secondhand engagement via `lit-review`; passing enumeration still never fires; `problem-statement-anchor` stays never-on-lit-review. `risk-ip`/`risk-bias` have no theme home вЂ” a contribution-level cluster promotes flag в†’ theme

**Data-collection method facets** (data-source axis вЂ” how the paper's **own evidence** was produced):
- `method-self-report` (humans tell you: questionnaires, interviews, focus groups, diaries) В· `method-mining` (artifacts measured: repos, PRs, commits, posts, logs, telemetry) В· `method-experiment` (controlled tasks with manipulation, lab or crowdsourced) В· `method-field-study` (deployment / case study observed in a real setting). **Apply all that fit** (mixed methods expected вЂ” a survey + observation study gets two). Applies to the paper's **own evidence production**: `lit-review` papers get **none** (evidence synthesized вЂ” the methods live in the primaries); absence = no empirical evidence produced (position papers, pure designs). Note `method-self-report` в‰  `survey-input` (method vs the finding's utility to the org survey вЂ” a mined study can be survey-input). **World-or-tool test:** results describing the *world* в†’ method facet; results describing only the *tool* (benchmarks, self-run tests over constructed corpora) в†’ no facet, that's `built-system` evaluation; a detector run over real repos gets `method-mining` only if the findings characterize the repos. Humans using the artifact always fires one: **assigned tasks / controlled conditions в†’ `method-experiment`; real work, own setting, natural use в†’ `method-field-study`** (study-site pilots & case-study evaluations = field-study); both can apply. **Subjects may be systems:** controlled studies of *third-party* tools/models whose findings characterize those systems = `method-experiment` (UDVHQ5HR); running **your own** system on a benchmark вЂ” even a standardized third-party one (UB2EVUFU on ProjDevBench) вЂ” is still tool-results, no facet (whose-properties test; benchmark-grade evidence = a `built-system` maturity note, staged `benchmark-evaluated` candidate)

## WORKFLOW
- **Thin input:** if only title+abstract (or degenerate text) is available в†’ tag conservatively (no form facets / `lit-review` unless explicit in the abstract) and flag `insufficient-input` instead of guessing.
- **Struggle signals в†’ check the core bar:** can't pick a primary / stretching a definition to fill the set = the paper likely doesn't belong at core в†’ `demote:context` and move on. Don't force tags to justify a disposition.
- **Facet checklist вЂ” run all seven questions on every paper** (misses cluster here; 42 tags exceed recall): **role** (problem-statement-anchor / survey-input / intro-framing / lit-review / counterpoint)? В· **form** (design-only | framework в†’ built-system в†’ adopted)? В· **scope** (general-ai? general-code? non-developer?)? В· **mode** (assistive / agentic)? В· **contribution** (steering? metrics? routing-signal?)? В· **risk-types** (security / quality / over-reliance / ip / bias вЂ” substantive treatment only, never intro-lists)? В· **method** (self-report / mining / experiment / field-study вЂ” own evidence only)?
- **PRIMARY** вЂ” one per paper. *Zotero entry only:* type `cal:human:primary:theme:<slug>`. In JSON output, the task block's bare-slug `primary_theme` field governs.
  - **Tie-breaker** when a paper spans adjacent themes (routeв†”control-surface, AI-checkв†”human-gate): primary = the theme carrying its **distinctive novelty**, not the standard scaffolding. (Hedwig: novelty = the dynamic-autonomy classifier в†’ `risk-routing`, not the check-in surface.)
  - **Altitude refinement:** when a broader and a more specific theme both fit, primary = the **most specific theme that still captures where the paper's main effort lives** (an AI-review vision whose energy goes into explanation design в†’ `oversight-explanation`). A specific mechanism that is merely one *component* of a broader contribution does NOT outrank the broader theme (budget-halving inside a verification architecture в†’ primary stays `ai-review`).
- **DEMOTE flag** (menu): `demote:context` вЂ” mis-scoped for core: **general-AI object В· steering-only solution В· pure tool benchmark** (capability benchmarking with no mechanism or deployable contribution; a benchmark *evaluating a contributed system* doesn't count) **В· secondary lit В· not operationalizable**. `general-code` alone is NOT a demote reason (it's the audit trail for kept-core transfers). Flag and keep going; batch-reviewed later. (`demote:discard` = false-positive that shouldn't be in the corpus at all.)
