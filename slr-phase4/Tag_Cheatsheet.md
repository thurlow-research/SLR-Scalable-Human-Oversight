# Theme-tagging cheat-sheet (Vibe Coding Governance SLR)

Tag on the **document contents**. **Multi-tag freely** ‚Äî a paper can carry several themes.
Tag on **membership** (the paper *contributes to* that theme's argument), **not mention**.
Pick **one PRIMARY theme** (the paper's home / where it'd be written up in depth; tie-breaker and lit-review override below).
**Steering exclusion:** shaping what gets *generated* ‚Äî prompts, specs, fine-tuning, **controlling the AI's inputs/context** ‚Äî is steering, NOT oversight; tag only the paper's inspection/comprehension/gating contribution.
**Plumbing ‚âÝ membership** (systems papers): tag only mechanisms the paper *argues about*, not everything its system happens to use (agents running test suites ‚âÝ `rules-based-checks`).
**Documented practice counts:** empirical evidence of practitioners *exercising* a stage's mechanism earns membership in that stage's theme. **Vision-paper floor:** a stage mentioned in one scenario sentence = mention, not membership.

## THEME TAGS (`theme:<slug>`)

**Problem ‚Äî quantify**
- `oversight-scaling-inversion` ‚Äî AI code is riskier yet *less* inspected; PRs auto-merged unreviewed; review is the bottleneck; burden piles on maintainers
- `ai-code-insecurity` ‚Äî empirical **security**-vulnerability evidence in AI code (CWEs, insecure-but-confident), **incl. hallucinated/poisoned dependencies in the output** (D4 ruling). Incident analyses & original vulnerability taxonomies count; secondhand rate citations alone don't
- `quality-debt` ‚Äî **non-security** quality decay: tech debt, complexity, smells, maintainability, breaking changes

**Limits of current oversight**
- `automation-bias` ‚Äî the *human* fails: over-trust, complacency, skill erosion, misses flaws even when warned. **Requires a CAPABLE human failing** (attention/trust); a human who *can't evaluate at all* (non-developer settings; failure survives priming+incentives) = competence gap, NOT bias (22JBEZNK). Competence-gap papers **stay core-eligible**: tag the remainder (explanation/mode/population facets) ‚Äî the missing theme is a staged candidate, not a demote signal
- `oversight-theater` ‚Äî review exists on paper but lacks authority/time/info to change the outcome (rubber-stamp, token HITL)

**Solution ‚Äî the Detect ‚Üí Triage ‚Üí Fix ‚Üí Escalate pipeline** *(gray-zone detectors ‚Äî rubric-grounded LLM-judge, "LLM writes tests then runs them" ‚Äî expect dual-tagging of both Detect themes)*
- `ai-review` ‚Äî [DETECT] AI *judges* the artifact (probabilistic, can hallucinate); incl. multi-agent / cross-model review, **its reliability limits**, and empirical evidence of practitioners *using* agents to validate agents' output
- `rules-based-checks` ‚Äî [DETECT] deterministic **grounded** checks: tests, static analysis/lint, symbolic exec, sandbox, classical formal-methods engines
- `formal-methods` ‚Äî [technique, COMPOSABLE] theorem proving / model checking / symbolic exec / autoformalization. Pair with the performer: AI does it ‚Üí `ai-review`+`formal-methods`; classical engine ‚Üí `rules-based-checks`+`formal-methods`; no performer (pure advocacy/position) ‚Üí `formal-methods` + the `intro-framing` *facet*
- `risk-routing` ‚Äî [TRIAGE] the **allocation decision** ‚Äî *which/whether/when* AI items reach a human & at what tier; the **smarts of surfacing** (signal + selection/tiering logic). NOT the human's control mechanism. **(= WHAT gets surfaced)** Error-condition handback ("agent stuck ‚Üí human") is NOT routing ‚Äî no triage decision ‚Üí `hitl-workflow` / `remediation-gating`. Signal must be **computed & producer-independent** ‚Äî model self-confidence is disqualified
- `remediation-gating` ‚Äî [FIX] governance of **autonomous** fixing (Jidoka: the system fixes *without per-fix human involvement*, kept safe by machinery) ‚Äî **content** gates (filter fix candidates) and **process** gates (bounded retries / budget-decay / convergence / stop-progression, fail-closed); not the repair technique itself. **REQUIRES an autonomous fix being overseen** ‚Äî a detect/publish gate with **no auto-fix** (VibeGuard) is NOT this (detector's enforcement side). Deciding when a **human** must engage on the fix path (risky fixes ‚Üí human) = `risk-routing` layered on top
  - *Anti-pattern note:* a human approving **every** fix is NOT remediation-gating ‚Äî review-everything is the unscalable anti-pattern (a paper *advocating* it ‚Üí `counterpoint` candidate). Such a design *introduces* automation-bias risk ‚Äî that's a critique to note, NOT `automation-bias` membership (membership requires the paper to *study* the human failure)
- `hitl-workflow` ‚Äî [ESCALATE] the human's **control surface** ‚Äî *how the human acts* once engaged: checkpoints, action guards, approval gates, bounded delegation. NOT what to surface. **Levers, not lenses** (comprehension/visibility tools ‚Üí `oversight-explanation`; a lever over the AI's *inputs* is steering). **Plan-gate rule:** a human gate over an AI-produced plan IS hitl-workflow when it's a **designed checkpoint in a lifecycle** (defined gate, authority to block); the same approval inside a conversational guide-then-"go do it" flow is steering. **(= HOW + WHEN the human acts)**
- `oversight-explanation` ‚Äî [ESCALATE¬∑info side] **help the human understand what the AI is doing** ‚Äî any system-provided or human-invoked support for understanding/judging AI output. Modes (illustrative, NOT exhaustive): *push* = decision-ready escalation handoff (background + options + recommendation + risks); *pull* = human-invoked visibility into what the AI is doing/using; *standing* = explanations attached to AI output that support the human's verdict. Lens, not lever (lever ‚Üí `hitl-workflow`)
- `agent-scope-drift` ‚Äî [cross-cutting] agents wander off-mandate / make unreviewed decisions; mechanisms that **detect/bound departure from intent**. Tag by the mechanism's **object**, not the actor's motivation: a panel checking code scope vs spec scope ‚Üí here + `ai-review`; retained control merely *motivated* by drift-worry ‚Üí `hitl-workflow` only

**Worked decomposition (multi-stage systems ‚Äî assign each mechanism by its object):** producer's own pre-submit checking/iterating = self-conformance, NO theme ¬∑ independent evaluation of the submitted artifact = detector (`rules-based-checks`/`ai-review` by epistemics; the *additional, producer-independent* checks are the oversight value; blocking on failure = the detector's enforcement edge) ¬∑ deciding which findings matter, e.g. by severity + cross-model agreement (a computed, producer-independent signal) ‚Üí `risk-routing` ¬∑ governing the ensuing **autonomous fix path** (filters, bounds, convergence, fail-closed stop ‚Äî no per-fix human) ‚Üí `remediation-gating` ¬∑ deciding when a fix must engage a **human** (risk tiers on the fix path) ‚Üí `risk-routing`

**Governance & policy**
- `org-governance` ‚Äî the *org-level governance apparatus* applied broadly: policy, **audit logging**, accountability, roles, maturity models, policy-as-code (internal; ‚âÝ a single pipeline framework)
- `regulatory-compliance` ‚Äî *external* law/standards: EU AI Act Art.14, NIST AI RMF, ISO 42001, liability, audit evidence

**Supply chain & provenance**
- `tooling-supply-chain` ‚Äî provenance/vetting of the AI **tools** (skills/MCP/agents); **+ attacks on the reviewer**. **Dependency risk *in generated code*** (hallucinated/poisoned packages the AI writes in) ‚Üí `ai-code-insecurity`, NOT here (D4 ruling). Excl. keyword false-positives (hardware trojans, classic supply-chain incidents)
- `provenance-auditability` ‚Äî traceability of AI **changes** so a human *can* review; auditable record; IP/licensing. Requires a **persistent record serving HUMAN reviewability/audit** ‚Äî a point-in-time "what's in use now" view ‚Üí `oversight-explanation`; persistence serving only agent coordination = plumbing

## FACET TAGS (functional role; orthogonal ‚Äî optional, apply if they fit)
- `problem-statement-anchor` ‚Äî a single committee-sit-up empirical stat **anchoring the OVERALL problem statement** (the scaling inversion / two-part frame) ‚Äî NOT a sub-argument's or population-specific headline number, however vivid. Never on `lit-review` (secondhand ‚Äî anchor the primaries)
- `survey-input` ‚Äî adoption / preference / RAI-priority finding useful for the org survey design; requires **substantive** findings ‚Äî one incidental adoption stat ‚âÝ membership
- `intro-framing` ‚Äî position / agenda / definitional paper that *names the gap* but **doesn't operationalize a mechanism**
- `lit-review` ‚Äî secondary literature (survey / review / meta-analysis) ‚Äî **systematic OR narrative**; test = evidence *synthesized from other papers, not produced*. **If `lit-review` applies, the primary MUST be the biggest-tent theme** = the theme under which the largest share of the synthesized evidence would be cited; ties ‚Üí the review's own framing question (never the most vivid/quantified section)
- `counterpoint` ‚Äî [role facet] the paper **argues against a prevailing position** (automation-maximalism, HITL value, oversight scaling ‚Äî any direction); note *what it opposes* in the rationale. Often pairs with `intro-framing`

**Artifact / evidence cluster** (composable; form ‚Üí maturity):
- `framework` ‚Äî a **technical framework / reference architecture integratable into a build pipeline** (whether or not built). ‚âÝ `theme:org-governance` (the org apparatus). A bare taxonomy/decision-model gets neither; **not a one-off point tool / bare point result** ‚Äî a focused single-concern architecture qualifies (VibeGuard, Hedwig). Test: *would someone adopt it as a reusable pipeline design?* Composes: `framework`+`built-system`+`adopted`.
- `design-only` ‚Äî the paper **specifies a mechanism in buildable detail** (components, formulas, thresholds, architecture) **but never credibly runs it**: no working implementation, no evaluation on the intended object. **A mock demo ‚âÝ built** ‚Äî a demo that fabricates the mechanism's core outputs (random/synthetic stand-ins for what it's supposed to compute) is still design-only (R4WJZBSF). Only for papers that *propose* something ‚Äî an empirical study with no proposed mechanism gets nothing. Mutually exclusive with `built-system`/`adopted`; composes with `framework`. ‚âÝ `intro-framing` (which specifies NO mechanism ‚Äî discriminator = buildable detail). A design-only *solution* paper ‚Üí check the core bar (unevaluated framework ‚Üí context)
- `built-system` ‚Äî the authors *implemented* it as a working system/tool/prototype ("and they built it")
- `adopted` ‚Äî used **outside research** (commercial / production / real org use), beyond a lab prototype/benchmark. Scarce, high-signal; absence = prototype/proposal

**Generation-mode scope pair** (which *setting* the paper studies; cut on **who initiates + the reviewable unit**, NOT tool location ‚Äî agents live in IDEs too):
- `assistive` ‚Äî **human-initiated, suggestion-granularity** generation (inline completion, **snippet-level** chat-paste): the human authors in the flow, accepts piece-by-piece ‚Äî a chat task returning a *complete artifact* is the tie-rule case below. Oversight surface = the *acceptance moment*.
- `agentic` ‚Äî **AI-initiated / AI-planned multi-step work** delivered at **artifact/PR granularity** for review. Oversight surface = the *gate*.
- Apply either or **both** (paper compares/spans modes); **neither** = mode unspecified / irrelevant to the paper's claim. **The pair describes the *generation* studied:** "uses agents" ‚âÝ `agentic`, "AI assists the human" ‚âÝ `assistive` ‚Äî AI only on the review/oversight side ‚Üí **neither** + consider `general-code`. **Tie-rule:** initiator vs reviewable unit disagree (human-prompted chat task ‚Üí complete artifact) ‚Üí **the reviewable unit dominates** (wholesale artifact = the gate = `agentic`).

**Scope / contribution / population flags:**
- `general-ai` ‚Äî [scope flag] governance/oversight is **general AI/LLM, not coding-specific** (model robustness, general RAI frameworks). Default (untagged) = coding-specific. Flags candidates for context.
- `steering` ‚Äî [contribution flag] the paper's proposed solution **or documented practice** operates on **generation** (prompts, specs, fine-tuning, shaping model inputs) rather than inspecting/gating the artifact, as a **substantive part of the contribution** ‚Äî NOT any incidental prompt-shaping component every AI system has. **Contribution, not topic.** Steering-**only** solution ‚Üí demote candidate
- `non-developer` ‚Äî [scope flag] the generating/overseeing human is **not a professional developer** (end-user / business user / citizen developer). Default (untagged) = professional devs
- `general-code` ‚Äî [scope flag] the oversight/review mechanism targets **code generally, not AI-generated code** (AI may sit on the *review* side only); transfers to our setting but wasn't developed/evaluated there. Default (untagged) = the overseen object is AI code

## WORKFLOW
- **Thin input:** if only title+abstract (or degenerate text) is available ‚Üí tag conservatively (no form facets / `lit-review` unless explicit in the abstract) and flag `insufficient-input` instead of guessing.
- **Struggle signals ‚Üí check the core bar:** can't pick a primary / stretching a definition to fill the set = the paper likely doesn't belong at core ‚Üí `demote:context` and move on. Don't force tags to justify a disposition.
- **Facet checklist ‚Äî run all five questions on every paper** (misses cluster here; 31 tags exceed recall): **role** (problem-statement-anchor / survey-input / intro-framing / lit-review / counterpoint)? ¬∑ **form** (design-only | framework ‚Üí built-system ‚Üí adopted)? ¬∑ **scope** (general-ai? general-code? non-developer?)? ¬∑ **mode** (assistive / agentic)? ¬∑ **contribution** (steering)?
- **PRIMARY** ‚Äî one per paper. *Zotero entry only:* type `cal:human:primary:theme:<slug>`. In JSON output, the task block's bare-slug `primary_theme` field governs.
  - **Tie-breaker** when a paper spans adjacent themes (route‚Üîcontrol-surface, AI-check‚Üîhuman-gate): primary = the theme carrying its **distinctive novelty**, not the standard scaffolding. (Hedwig: novelty = the dynamic-autonomy classifier ‚Üí `risk-routing`, not the check-in surface.)
  - **Altitude refinement:** when a broader and a more specific theme both fit, primary = the **most specific theme that still captures where the paper's main effort lives** (an AI-review vision whose energy goes into explanation design ‚Üí `oversight-explanation`). A specific mechanism that is merely one *component* of a broader contribution does NOT outrank the broader theme (budget-halving inside a verification architecture ‚Üí primary stays `ai-review`).
- **DEMOTE flag** (menu): `demote:context` ‚Äî mis-scoped for core: **general-AI object ¬∑ steering-only solution ¬∑ pure tool benchmark** (capability benchmarking with no mechanism or deployable contribution; a benchmark *evaluating a contributed system* doesn't count) **¬∑ secondary lit ¬∑ not operationalizable**. `general-code` alone is NOT a demote reason (it's the audit trail for kept-core transfers). Flag and keep going; batch-reviewed later. (`demote:discard` = false-positive that shouldn't be in the corpus at all.)
